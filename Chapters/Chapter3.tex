\section{Арга зүй}
\label{sec:methodology}

Тус судалгаа нь crowdsourced хүргэгчдийн сүлжээнд суурилсан B2B хүргэлтийн платформ хөгжүүлэхэд чиглэсэн \textbf{практик судалгаа} бөгөөд \textbf{загварчлал, хөгжүүлэлт, туршилт} гэсэн гурван үе шаттайгаар Agile аргачлалаар~\cite{agile2001manifesto} хэрэгжсэн. Iteration бүрд шинэ чадамж нэмж, шалгаж, засах давталттай бүтэцтэй явсан.

Хөгжүүлэлтийн үе шатууд:
\begin{enumerate}
  \item \textbf{Шаардлагын шинжилгээ}: бизнесийн логик, хэрэглэгчийн хэрэгцээг тодорхойлох
  \item \textbf{Архитектурын загварчлал}: системийн бүтэц, өгөгдлийн загвар, технологийн стек сонгох
  \item \textbf{Өгөгдлийн сангийн хөгжүүлэлт}: хүснэгтүүд, Row Level Security (RLS) бодлого, триггер, RPC функцүүд
  \item \textbf{Backend тохиргоо}: Supabase authentication, file storage, realtime холболт
  \item \textbf{Frontend хөгжүүлэлт}: мобайл апп (React Native/Expo), веб апп (Next.js)
  \item \textbf{Нэгтгэлийн туршилт}: бүрэлдэхүүн хэсгүүдийн хоорондын холболт шалгах
  \item \textbf{Гүйцэтгэлийн туршилт}: зэрэгцээ хандалт, аюулгүй байдлын тестүүд
\end{enumerate}

\textbf{Хүргэгчийн мобайл апп} (React Native/Expo) нь хүргэгчдэд зориулсан гар утасны аппликейшн бөгөөд бүртгэл, KYC баталгаажуулалт, нээлттэй даалгавар харах, даалгавар авах, хүргэлт гүйцэтгэх үндсэн үйлдлүүдийг хийнэ.

\subsubsection{Multi-tenant архитектур}

Систем нь олон байгууллагад зориулагдсан multi-tenant архитектуртай. Байгууллага бүр өөрийн бүтээгдэхүүн, захиалга, даалгаврын өгөгдлийг бүрэн тусгаарлагдсан хэлбэрээр удирдана. Tenant isolation-ыг PostgreSQL-ийн Row Level Security (RLS) бодлого хангаж, өгөгдлийн сангийн түвшинд баталгаажуулна. Тухайн хэрэглэгчийн \texttt{org\_id}-г RPC функцээр тодорхойлж, RLS бодлогод ашиглана.

\subsection{Системийн архитектур}
\label{subsec:system_architecture}

\subsubsection{Системийн бүрэлдэхүүн хэсгүүдийн тайлбар}

Системийн ерөнхий архитектур нь хоёр клиент аппликейшн (хүргэгчийн React Native/Expo апп болон борлуулагчийн Next.js 16 веб апп), нэгдсэн Supabase backend, болон тэдгээрийг холбосон протокол, үйлчилгээнүүдээс бүрдэнэ. Клиентүүд нь Supabase API gateway-ээр (HTTPS/REST болон WebSocket) дамжуулан backend-тэй холбогдоно.

Supabase платформын үндсэн үйлчилгээнүүд нь дараах байдалтай:
\begin{itemize}
  \item \textbf{Auth} — JWT токен үүсгэнэ;
  \item \textbf{PostgreSQL} — RLS бодлоготой өгөгдлийн сан;
  \item \textbf{Storage} — KYC баримт хадгална;
  \item \textbf{Realtime} — WebSocket-д суурилсан мэдэгдэл илгээнэ;
  \item \textbf{RPC функцүүд} — \texttt{SECURITY DEFINER} эрхтэй бизнесийн логик;
  \item \textbf{Edge Functions} — имэйл болон ePOD OTP илгээнэ;
  \item \textbf{Триггерүүд} — төлөв шилжилт, орлогын автомат бүртгэл.
\end{itemize}
Эдгээр үйлчилгээ хоорондоо JWT token validation, trigger-RPC холболт, Realtime channel-аар уялдана.

\subsubsection{BaaS-д суурилсан архитектур (BaaS-mediated architecture)}
\label{subsubsec:baas_architecture}

Уламжлалт \textit{three-tier} загварт бизнесийн логикийг тусдаа application server (Node.js/Express, Spring Boot) дээр байрлуулдаг. Харин тус систем нь тусдаа backend сервер ажиллуулдаггүй. Supabase-ийн BaaS платформ бүх backend үйлчилгээг зуучилдаг. Иймд архитектурыг \textbf{BaaS-mediated architecture}~\cite{jonas2019berkeleyserverless} гэж нэрлэх нь илүү зөв. API давхарга PostgREST-ээр автоматаар үүсгэгдэнэ. Бизнесийн логик нь RPC, триггер, Edge Functions-д тархсан.

\textbf{Client layer} нь хэрэглэгчийн интерфейс, навигаци, маягтын шалгалт, state management (React Context) зэрэг frontend логикийг хариуцна. Мобайл апп нь React Navigation ашиглан дэлгэцүүдийг удирдаж, веб апп нь Next.js-ийн App Router болон middleware ашиглана. Хоёр клиент хоёулаа \texttt{@supabase/supabase-js} SDK-аар дамжуулан BaaS давхаргатай HTTPS болон WebSocket протоколоор холбогдоно.

\textbf{BaaS layer} нь системийн API давхаргын үүргийг гүйцэтгэдэг Supabase платформ юм. Түүний хамгийн гол бүрэлдэхүүн нь \textbf{PostgREST} — PostgreSQL schema-г уншиж, бүх хүснэгт, view, функцийг автоматаар RESTful endpoint болгон хувиргах open source сервер. Ингэснээр клиент нь \texttt{POST /rest/v1/rpc/claim\_delivery\_task} зэрэг URL-аар шууд өгөгдлийн сангийн функцийг дуудах боломжтой болно. API-гийн дэргэд GoTrue (JWT authentication), Realtime (PostgreSQL \texttt{WAL}-ээс WebSocket мэдэгдэл), Storage (файл), Edge Functions (Deno runtime дээр ажилладаг имэйл илгээх гэх мэт гадаад integration) зэрэг туслах үйлчилгээнүүд ажиллана.

\textbf{Data layer} нь PostgreSQL өгөгдлийн сан, Row Level Security (RLS), триггерүүд, \texttt{SECURITY DEFINER} RPC функцүүдийг багтаана. Бизнесийн логикийн дийлэнх хэсэг — permission check, state machine, зэрэгцээ хандалтын \texttt{FOR UPDATE SKIP LOCKED} хамгаалалт — нь энэ давхаргад PL/pgSQL кодоор хэрэгжсэн. Энэ нь BaaS-mediated архитектурын өвөрмөц шинж бөгөөд клиент талын логикоос хамааралгүй аюулгүй ажиллана.

\subsubsection{Schema drift-ээс хамгаалах стратеги}
\label{subsubsec:schema_drift}

BaaS-mediated архитектурын гол сөрөг тал нь клиент аппликейшн PostgREST-ээр дамжуулан schema-тай шууд холбогддог явдал юм. Хүснэгтийн баганын нэр, төрөл өөрчлөгдөх бүрт клиент код зохицож шинэчлэгдэх шаардлагатай. Уг хамаарлыг \textit{schema drift} гэдэг — database-ийн хэлбэр ба клиентийн таамагласан хэлбэрийн хоорондын зөрүү~\cite{fowler2003evolutionary}. Гараар удирдвал runtime-д \texttt{column ``\ldots'' does not exist} алдаа гарч, production-д хагарал үүсгэх эрсдэлтэй.

Уг эрсдэлийг эхнээс нь хаахын тулд дараах \textbf{schema drift prevention strategy}-г төсөлд хэрэгжүүлэв:

\begin{enumerate}
  \item \textbf{Автомат TypeScript type үүсгэлт.} Supabase CLI-ийн \texttt{supabase gen types typescript --linked} команд нь production-той холбогдсон (linked) төслийн schema-г уншиж, бүх хүснэгт, view, enum, RPC функцийн signature-ийг нэгтгэсэн \texttt{types/database.ts} файлыг үүсгэнэ. Үүнд баганын нэр, төрөл, NOT NULL шинжийг compile-time-д илэрхийлэх \texttt{Database[``public''][``Tables''][``orders''][``Row'']} гэх мэт type-ууд багтана.

  \item \textbf{CI pipeline integration.} Уг командыг GitHub Actions workflow-д шат болгон оруулсан. Pull request нээгдэх бүрт CI нь linked төслөөс шинэ \texttt{database.ts} үүсгэж, commit-логдсон хувилбартай харьцуулна. Зөрүү илэрвэл build унаж, хөгжүүлэгчийг schema migration-ы дараа type-ийг дахин үүсгэж commit-лохыг шаарддаг.

  \item \textbf{Compile-time contract enforcement.} Клиент код нь үүсгэгдсэн типийг импортлоод Supabase client-ийг \texttt{createClient<Database>(url, key)} гэх хэлбэрээр параметерждэг. Ингэснээр \texttt{supabase.from(``orders'').select(``customer\_id, total'')} хэлбэрийн дуудлага бүр статик шалгалтад орж, байхгүй багана эсвэл буруу төрөл дуудахад TypeScript compiler-ээр runtime-аас өмнө илэрнэ.

  \item \textbf{Migration-д суурилсан schema удирдлага.} Өгөгдлийн сангийн бүтцийг зөвхөн \texttt{supabase/migrations/} хавтасны SQL migration файлаар өөрчлөх дүрэм тогтоосон. Supabase Studio эсвэл \texttt{psql}-ээр шууд гараар өөрчлөхийг хориглож, ингэснээр schema-ийн түүх git-ээр бүрэн хянагддаг болгосон.
\end{enumerate}

Уг стратегийн үр дүнд client–server contract нь compile-time-д баталгаажиж, schema өөрчлөлт runtime-д биш build-time-д илрэнэ. Гэсэн хэдий ч coupling-ийн үндсэн асуудлыг бүрэн шийдэхгүй тул BFF (Backend-for-Frontend) давхарга нэмэх нь цаашдын ажлын чухал чиглэл юм.

\subsubsection{Authentication архитектур}
\label{subsubsec:auth_architecture}

Систем нь хоёр ялгаатай authentication арга ашиглана — веб аппд серверийн талын (SSR) authentication, мобайл аппд клиент талын authentication.

\textbf{Борлуулагчийн веб апп (Next.js SSR)}

\begin{enumerate}
  \item Хэрэглэгч имэйл, нууц үгээр нэвтэрнэ
  \item Supabase Auth сесс үүсгэж, JWT токен олгоно
  \item Токен серверийн талын cookie-д хадгалагдана
  \item Next.js middleware нь хүсэлт бүрд сессийг шалгаж, токен шинэчилнэ
  \item Хамгаалагдсан замууд (\texttt{/dashboard}, \texttt{/tasks}, \texttt{/orders} г.м.) нэвтрээгүй бол нэвтрэх хуудас руу чиглүүлнэ
  \item Борлуулагчийн layout нь хэрэглэгчийн эрхийн түвшинг (role) шалгаж, зөвхөн зөвшөөрөгдсөн эрхтэй хэрэглэгчийг нэвтрүүлнэ
\end{enumerate}

\textbf{Хүргэгчийн мобайл апп (React Native)}

\begin{enumerate}
  \item Хэрэглэгч имэйл, нууц үгээр нэвтэрнэ (Supabase Auth)
  \item Сесс AsyncStorage-д хадгалагдана (оффлайн хандалтын дэмжлэг)
  \item CourierAuthContext нь authentication төлвийг нэгдсэн байдлаар удирдана
  \item Апп эхлэхэд \texttt{restoreSession()} ажиллаж, хүчинтэй сессийг сэргээнэ
  \item Хүргэгчийн эрхийн түвшин (courier), батламжийн төлөв (pending/approved/blocked) шалгагдана
  \item Батлагдаагүй бол KYC шаардлагатай эсвэл хүлээж буй дэлгэц харуулна
  \item Төлөв ``pending'' бол 60 секунд тутам автоматаар шинэчилнэ (админ батлахад шууд шинэчлэгдэнэ)
\end{enumerate}

\subsection{Системийн хэрэглэгчид}
\label{subsec:system_users}

Платформ нь олон талт зах зээлийн зарчимд суурилсан бөгөөд дараах гурван үндсэн оролцогчийн харилцан үйлчлэлд тулгуурлана.

\subsubsection{Байгууллага (Борлуулагч)}
Байгууллага нь хүргэлтийн хэрэгцээ шаардлага үүсгэдэг \textbf{эрэлтийн тал} юм. Платформ нь дараах төрлийн байгууллагуудын хүргэлтийг дэмждэг:
\begin{itemize}
  \item \textbf{Ресторан}: хоолны хүргэлт
  \item \textbf{Дэлгүүр}: жижиглэн худалдааны бүтээгдэхүүн
  \item \textbf{Эмийн сан}: эм, эрүүл мэндийн бүтээгдэхүүн
  \item \textbf{Агуулах}: том хэмжээний ачаа хүргэлт
\end{itemize}

Байгууллагын хэрэглэгчийн үндсэн чадамж:
\begin{enumerate}
  \item Веб аппаар байгууллагын бүртгэл үүсгэх, мэдээлэл оруулах
  \item Бүтээгдэхүүний каталог удирдах (нэмэх, засах, устгах)
  \item Хэрэглэгчээс ирсэн захиалга хүлээн авах, боловсруулах
  \item Хүргэлтийн даалгавар үүсгэх, хүргэлтийн хураамж тогтоох
  \item Даалгаврыг зах зээлд нийтлэх (нийтлэгдсэн төлөвт шилжүүлэх)
  \item Хүргэлтийн явц хянах, гүйцэтгэлийг баталгаажуулах
\end{enumerate}

Борлуулагч нь веб аппликейшнд дараах хуудсуудыг ашиглана:
\begin{itemize}
  \item \textbf{Хянах самбар (Dashboard)}: гол үзүүлэлтүүд (KPI) — нийт даалгавар, идэвхтэй даалгавар, дууссан даалгавар, орлогын мэдээлэл, өнөөдөр ба өчигдрийн харьцуулалт, бодит цагийн холболтын төлөв
  \item \textbf{Бүтээгдэхүүн}: бүтээгдэхүүний каталог удирдлага (нэмэх, засах, устгах, боломжтой/боломжгүй хугацаа тохируулах)
  \item \textbf{Даалгавар}: хүргэлтийн даалгавар үүсгэх, жагсаалт (төлвөөр шүүх, хуудаслалт), дэлгэрэнгүй, олноор нийтлэх (bulk publish), төлвийн шинэчлэл
  \item \textbf{Захиалга}: захиалгын удирдлага — гурван табтай (``Үйлдэл шаардлагатай'' — төлбөр хийгдсэн захиалгууд, ``Идэвхтэй'' — бэлтгэж буй/бэлэн, ``Түүх'' — дууссан/цуцлагдсан). Захиалга бүрийн дэлгэрэнгүй мэдээлэл, бараа задаргаа, төлвийн шинэчлэл
  \item \textbf{Санхүүгийн тайлан (Financials)}: өнөөдрийн орлого, өчигдрийн орлого, нийт орлого, хүргэлт бүрийн орлогын задаргаа
  \item \textbf{Аналитик (Analytics)}: даалгаврын гүйцэтгэлийн шинжилгээ, орлогын чиг хандлага (Recharts график сантай)
  \item \textbf{Тохиргоо}: байгууллагын мэдээлэл засах, ажлын цагийн тохиргоо
\end{itemize}

\subsubsection{Хүргэгч}
Хүргэгч нь \textbf{нийлүүлэлтийн тал} бөгөөд бие даасан гүйцэтгэгч юм. Хүргэгч нь гар утасны аппликейшн ашиглан системтэй харилцана.

Хүргэгчийн бүртгэлийн болон ажиллагааны үе шатууд:
\begin{enumerate}
  \item \textbf{Бүртгүүлэх}: мобайл аппаар имэйл, нууц үгээр бүртгүүлнэ (эрхийн түвшин автоматаар хүргэгч болж тохируулагдана)
  \item \textbf{Баталгаажуулалтын баримт илгээх}: иргэний үнэмлэхний зураг (урд/ар), тээврийн хэрэгслийн бүртгэл, хувийн зураг
  \item \textbf{Админ батламж хүлээх}: баримт илгээсний дараа ``илгээсэн'' төлөвт шилжиж, админ шалгана
  \item \textbf{Батлагдсан}: нээлттэй даалгавар харах, авах эрхтэй болно
  \item \textbf{Даалгавар авах}: зах зээлээс даалгавар сонгон авна
  \item \textbf{Хүргэлт гүйцэтгэх}: бараа авах $\rightarrow$ хэрэглэгчид хүргэх
  \item \textbf{Орлого хянах}: гүйцэтгэсэн хүргэлтийн орлогыг харна
\end{enumerate}

Хүргэгч мобайл аппын үндсэн хэсгүүд:
\begin{itemize}
  \item \textbf{Нээлттэй даалгавар}: зах зээлийн нээлттэй даалгаврын жагсаалт
  \item \textbf{Нүүр}: хүргэгчийн хянах самбар, тойм мэдээлэл
  \item \textbf{Захиалга}: идэвхтэй болон хуваарилагдсан захиалгууд
  \item \textbf{Орлого}: хүргэлтийн орлогын түүх
  \item \textbf{Профайл}: хувийн мэдээлэл засах
\end{itemize}

\subsubsection{Админ}
Админ нь системийн ерөнхий удирдлага хийх давуу эрхтэй хэрэглэгч юм.
\begin{enumerate}
  \item Хүргэгчийн баталгаажуулалтын баримтыг шалгах, батлах эсвэл татгалзах
  \item Хэрэглэгчийн эрхийн түвшин өөрчлөх
  \item Хүргэгчийн эрх хаах (хоригдсон төлөвт шилжүүлэх)
  \item Системийн ерөнхий хяналт хийх
\end{enumerate}

%-------------------------------------------------------------------------------
\subsubsection{Нэвтрэх ба authentication систем}

Систем нь Supabase Auth-ийн дэд бүтэц дээр суурилсан. Хэрэглэгч бүртгүүлэх үед апп-аас хамаарч эрх автомат тохируулагдана (хүргэгчийн апп — хүргэгч, веб — борлуулагч); түвшин өөрчлөх зөвхөн админд боломжтой. Техникийн хэрэгжүүлэлтийг (JWT, сесс, триггер) \ref{subsubsec:auth_architecture}-р хэсэгт тодорхойлсон.

Хүргэгчийн KYC процесс нь \textbf{Хүлээгдэж буй} → \textbf{Баримт илгээсэн} → \textbf{Батлагдсан} (эсвэл \textbf{Хоригдсон}) гэсэн төлвийн шилжилтээр явагдана. Илгээх баримт: иргэний үнэмлэх (урд/ар), тээврийн хэрэгслийн бүртгэл, хувийн зураг, тээврийн хэрэгслийн мэдээлэл (Supabase Storage-д RLS-ээр хамгаалагдан хадгалагдана).

%-------------------------------------------------------------------------------
\subsubsection{Байгууллагын удирдлага}

\textbf{Олон байгууллагын өгөгдлийн тусгаарлалт}

Олон байгууллага нэг системд зэрэг ажиллах боловч байгууллага бүрийн өгөгдөл бүрэн тусгаарлагдсан байна (тусгаарлалтын RLS бодлогуудыг \ref{subsubsec:security_architecture}-р хэсэгт тодорхойлсон).
\begin{itemize}
  \item Байгууллагын хүснэгт: дугаар, нэр, төрөл (ресторан, дэлгүүр, эмийн сан, агуулах), үүсгэсэн огноо
  \item Хэрэглэгч-байгууллагын холбоос: профайл дахь байгууллагын дугаар (хүргэгч, хэрэглэгчид хамаарахгүй)
\end{itemize}

\textbf{Байгууллагын бүртгэлийн урсгал (Onboarding)}

Шинэ борлуулагч бүртгүүлэх үед дараах үе шатуудыг дамжина:
\begin{enumerate}
  \item Имэйл, нууц үгээр хэрэглэгчийн бүртгэл үүсгэх
  \item Байгууллагын мэдээлэл оруулах — нэр, төрөл (ресторан, дэлгүүр, эмийн сан, агуулах), хаяг, холбоо барих мэдээлэл
  \item Амжилттай бүртгэл дууссаны дараа хянах самбар руу шилжинэ
\end{enumerate}

Нууц үг мартсан тохиолдолд имэйлээр нууц үг сэргээх (forgot password / reset) боломж хэрэгжсэн. Supabase-ийн сэргээлтийн линк илгээх механизмыг ашиглав.

\textbf{Бүтээгдэхүүний каталог удирдлага}

Байгууллага өөрийн бүтээгдэхүүнүүдийг нэмэх, засах, устгах, харах үйлдлээр удирдана. Бүтээгдэхүүн бүр тухайн байгууллагын дугаартай холбоотой бөгөөд Row Level Security (RLS)-ээр тусгаарлагдсан.

%-------------------------------------------------------------------------------
\subsubsection{Захиалгын удирдлагын систем}

Захиалгын амьдралын мөчлөг нь \textbf{Төлбөр хүлээгдэж буй} $\rightarrow$ \textbf{Төлбөр хийгдсэн} $\rightarrow$ \textbf{Бэлтгэж буй} $\rightarrow$ \textbf{Хүргэлтэд бэлэн} $\rightarrow$ \textbf{Бараа авсан} $\rightarrow$ \textbf{Хүргэгдсэн} шилжилтээр явагдана; тодорхой төлвүүдээс \textbf{Цуцлагдсан} төлөвт шилжих боломжтой. Төлбөрийн баталгаажуулалт нь ӨС-гийн триггерээр нэгдмэл байдлаар хийгдэж, төлбөрийг гараар өөрчлөх боломжгүй.

Борлуулагчийн веб апп нь захиалгыг гурван табтай бүтэцэд харуулна: ``Үйлдэл шаардлагатай'' (төлбөр хийгдсэн боловч бэлтгэгдээгүй), ``Идэвхтэй'' (бэлтгэж буй/бэлэн), ``Түүх'' (хүргэгдсэн/цуцлагдсан). Захиалга бүрийг дэлгэрэнгүй харах, баталгаажуулалтын цонхоор төлөв шилжүүлэх боломжтой.

%-------------------------------------------------------------------------------
\subsubsection{Хүргэлтийн даалгаврын систем}

Энэ модуль нь системийн \textbf{үндсэн шинэлэг шийдэл} болох зах зээлд суурилсан хүргэлтийн хуваарилалтыг хэрэгжүүлнэ. Байгууллага хүргэлтийн даалгавар үүсгэхдээ авах/хүргэх байршил (хаяг, координат), хүргэлтийн хураамж, тайлбарыг оруулна; даалгаврыг ``нийтлэгдсэн'' төлөвт шилжүүлснээр зах зээлд бүх батлагдсан хүргэгчид харагдана.

Даалгаврын төлвийн шилжилт: \textbf{Ноорог} $\rightarrow$ \textbf{Нийтлэгдсэн} $\rightarrow$ \textbf{Хуваарилагдсан} $\rightarrow$ \textbf{Бараа авсан} $\rightarrow$ \textbf{Хүргэгдсэн} (ePOD хүлээж буй) $\rightarrow$ \textbf{Дууссан} (орлого бүртгэгдэнэ); тусдаа \textbf{Цуцлагдсан} төлөв байна. Олон хүргэгч нэг даалгаврыг зэрэг авахыг оролдоход \texttt{claim\_delivery\_task} функц \ref{subsubsec:rpc_architecture}-р хэсэгт тайлбарласан атомик түгжээний механизмаар зөвхөн нэгийг нь амжилттай болгоно. Төлөв өөрчлөгдөх бүрт холбогдсон талуудад Supabase Realtime мэдэгдэл очно.

%-------------------------------------------------------------------------------
\subsubsection{Цахим хүргэлтийн баталгаа (ePOD) ба орлогын систем}
\label{subsubsec:epod_system}

ePOD нь хүргэлт амжилттай гүйцэтгэгдсэнийг нотлох OTP суурилсан дижитал баталгаажуулалт юм. Ажиллагааны урсгал:

\begin{enumerate}
  \item Хүргэгч ``хүргэгдсэн'' төлөвт шилжүүлнэ
  \item Систем 6 оронтой OTP үүсгэж хүлээн авагчид илгээнэ
  \item Хүлээн авагч кодыг хүргэгчид хэлж, хүргэгч мобайл аппд оруулна
  \item Амжилттай бол даалгавар ``дууссан'' төлөвт шилжиж, хүргэгчийн орлого автоматаар \texttt{courier\_earnings} хүснэгтэд бүртгэгдэнэ
\end{enumerate}

Техникийн хэрэгжүүлэлтийн дэлгэрэнгүй мэдээллийг (RPC функцүүд, аюулгүй байдлын механизм) \ref{subsubsec:rpc_architecture}-р хэсэгт тодорхойлсон болно.

Хүргэгч мобайл аппын ``Орлого'' хэсгээр нийт орлого, хүргэлт бүрийн задаргаа, дууссан хүргэлтийн түүхийг харна.

\subsubsection{Хүснэгтүүдийн тайлбар}

\begin{itemize}
  \item \textbf{auth.users, profiles, organizations}: Хэрэглэгчийн нэвтрэлт, профайл (role, status) болон олон байгууллагын (multi-tenant) тусгаарлалт.
  \item \textbf{orders, payments}: Захиалга ба төлбөр (qpay / stripe / bank / cash).
  \item \textbf{locations, delivery\_tasks, task\_assignments}: Авах/хүргэх байршил, даалгаврын үндсэн хүснэгт (status: draft $\rightarrow$ published $\rightarrow$ assigned $\rightarrow$ picked\_up $\rightarrow$ delivered $\rightarrow$ completed), хуваарилалтын audit trail.
  \item \textbf{courier\_kyc, courier\_earnings}: Хүргэгчийн KYC баримт ба ePOD амжилттай дууссан үед автомат бүртгэгдэх орлого.
  \item \textbf{epod\_otps}: Хүргэлтийн OTP код — \texttt{generate\_epod\_otp()} / \texttt{verify\_epod\_otp()} функцээр удирдагдана.
\end{itemize}

\subsubsection{Өгөгдлийн бүрэн бүтэн байдлын дүрмүүд ба \texttt{available\_tasks} VIEW}

Бүх хүснэгт uuid төрлийн primary key, foreign key constraint, enum-д суурилсан check constraint ашиглана. Уникал хязгаарлалтууд: \texttt{payments.order\_id}, \texttt{delivery\_tasks.order\_id}, \texttt{courier\_kyc.courier\_id}. Олон байгууллагын тусгаарлалтыг \texttt{org\_id} талбараар RLS бодлогоор, зэрэгцээ хандалтыг \texttt{FOR UPDATE SKIP LOCKED} түгжээгээр хангана~\cite{postgresql2024locking, kleppmann2017ddia}. Захиалга устгахад холбоотой \texttt{order\_items}, \texttt{payments}, \texttt{delivery\_tasks} каскадаар устгагдана.

Нээлттэй даалгаврууд нь \texttt{available\_tasks} VIEW-ээр зах зээлд харагдана — шүүлт нь \texttt{status = 'published'} ба \texttt{courier\_id IS NULL}, мөн хүргэгч нь захиалагч байгууллагад харьяалагдахгүй байх нөхцөлийг хангана.
