\noindent{\headingfont 1. Удиртгал}

Монгол Улсын жижиг, дунд бизнесийн байгууллагууд (ЖДБ) хоорондоо бараа захиалах, хүргэх үйл явцаа одоо хүртэл гол төлөв утас, мессеж, цаасан бичгээр гүйцэтгэсээр байна. Ийм гараар зохицуулдаг хэлбэр нь цаг алдагдуулж, зохион байгуулалтын алдаа үүсгэж, явцыг хянахад төвөгтэй болгодог. Монголын зах зээлд B2B хүргэлтийн нэгдсэн шийдэл хараахан төлөвшөөгүй бөгөөд одоогийн тоглогчид голдуу dispatcher-т тулгуурласан нэг талтай загвартай хэвээр байна.

Судалгааны зорилго нь:
\begin{itemize}
    \item Operator-гүй, multi-tenant B2B хүргэлтийн платформыг Supabase (PostgreSQL + PostgREST + RLS) технологи дээр боловсруулах,
    \item Concurrency хамгаалалтыг database түвшинд шийдэж баталгаажуулах,
    \item Multi-tenant data isolation-ыг нотлох,
    \item ePOD OTP верификацыг аюулгүй болгох,
    \item Туршилтаар системийн performance-ийг үнэлэх.
\end{itemize}

Судалгааны асуултууд (RQ) нь дараах гурван чиглэлд төвлөрсөн: RQ1 — FCFS claim загварын concurrency correctness; RQ2 — RLS-ийн multi-tenant isolation найдвартай байдал; RQ3 — BaaS-mediated архитектурын production viability. Эдгээрийг concurrency test, load test, security penetration test, functional E2E test гэсэн 4 төрлийн туршилтаар хэмжив.

\noindent{\headingfont 2. Судалгааны сэдвийн онол, өнөөгийн байдал}

B2B хүргэлтийн платформын онолын суурь нь таван үндсэн чиглэлд тулгуурласан: last-mile логистик, two-sided marketplace, multi-tenant isolation, concurrency control, BaaS архитектур.

Last-mile хүргэлт нь нийт тээврийн зардлын 13--75\%-ийг эзэлдэг тул crowdsourced загвар нь уламжлалт өөрийн парктай системийг орлох шинэ чиг хандлага болсон. Two-sided marketplace онолоор (Rochet \& Tirole 2003) B2B платформын гол саад нь liquidity тул cold-start шатанд FCFS загвар давуу талтай.

Multi-tenant өгөгдлийн тусгаарлалтыг Chong (2006), Bezemer \& Zaidman (2010) нар database түвшинд хэрэгжүүлэхийг зөвлөсөн. Үүнийг PostgreSQL Row Level Security (RLS) нь хүснэгтийн мөр бүрд \texttt{USING}/\texttt{WITH CHECK} policy хавсарган хангана.

Concurrency control-ыг Bernstein (1987), Kleppmann (2017) нар pessimistic/optimistic хоёр хандлагад ангилсан. Write-heavy workload-д pessimistic илүү найдвартай тул PostgreSQL-ийн \texttt{SELECT ... FOR UPDATE SKIP LOCKED} нь race condition-ыг 0 магадлалаар шийднэ.

Харьцуулсан судалгаанд Uber Freight (ML-д суурилсан enterprise платформ) болон Papa Logistics (dispatcher-based Монголын платформ) хоёрыг авч үзсэн. Эдгээрээс dispatcher bottleneck, data isolation сул, audit trail хязгаарлагдмал, API integration сул гэсэн 4 архитектурын зөрүүг тодорхойлсон.

\noindent{\headingfont 3. Судалгааны арга зүй}

Судалгааг Design Science Research арга барилаар 3 үе шатаар (загварчлал → хөгжүүлэлт → туршилт) Agile давталттай хэрэгжүүлэв. Системийн архитектурын гол онцлог нь \textbf{BaaS-mediated architecture} бөгөөд уламжлалт three-tier биш, Supabase BaaS платформоор бүх backend үйлчилгээг зуучилсан гурван давхаргат бүтэц юм (Зураг~\ref{fig:tovchlol_arch}).

Client layer нь хэрэглэгчийн интерфэйсийг хариуцна. BaaS layer нь Supabase платформоор API-г автомат үүсгэж, JWT authentication, realtime WebSocket, storage болон RPC дуудлагыг нэгтгэнэ. Data layer нь PostgreSQL + RLS + триггер + \texttt{SECURITY DEFINER} функцээр бизнесийн логикийг database дотор хэрэгжүүлнэ.

Өгөгдлийн түвшинд 11 хүснэгт бүхий schema зохион байгуулж, multi-tenant isolation-ыг \texttt{org\_id} талбар дээр суурилсан RLS-ээр, зэрэгцээ claim-ийг \texttt{FOR UPDATE SKIP LOCKED} lock-оор хангасан. Бизнесийн логикийн гол функцүүд: \texttt{claim\_delivery\_task} (атомик байдлаар даалгавар оноох), \texttt{verify\_epod\_otp} (bcrypt харьцуулалтаар OTP баталгаажуулах), мөн автомат төлөв шилжилт, орлого бүртгэлийн триггерүүд.

Proof of Delivery (ePOD)-г 6 оронтой OTP код (bcrypt hash, 10 мин expiry, 5 оролдлогын хязгаар) хэлбэрээр хэрэгжүүлсэн. Client-server contract-ыг хамгаалах schema drift prevention strategy-г Supabase CLI-ийн auto-generated TypeScript type + CI шатны харьцуулалтаар хангасан.

\textbf{Системийн үндсэн урсгал.} Бүртгэлээс орлого бүртгэл хүртэлх үйл явцыг 8 алхамт дараалалд зохион байгуулсан (Зураг~\ref{fig:tovchlol_flow}). Энэ урсгал нь веб апп (1–4 алхам), мобайл апп (5–7 алхам), database триггер (8 алхам) гэсэн 3 бүрэлдэхүүнийг нэг гинжид уялдуулсан.

\noindent{\headingfont 4. Судалгааны үр дүн, нэгтгэсэн дүгнэлт}

\textbf{Concurrency test.} $N \in \{10, 50, 100\}$ хүргэгч нэг даалгаврыг зэрэг claim хийх 30 сорилын туршид \textbf{давхардсан claim нэг ч удаа гараагүй} (100\% амжилт). 100 зэрэгцээ хандалт 512~ms-д шийдэгдсэн нь O($n$) шугаман хэмжээтэй, deadlock-гүй pessimistic locking-ийн онолын хүлээгдсэн зан төлөвтэй нийцнэ.

\textbf{Load test.} k6-аар 100 VU, 30 мин ачаалал өгөхөд бүх endpoint-ийн p95 latency 2 секундын зорилтот хязгаарт багтсан:

\begin{multicols*}{2}

\textbf{Security penetration test.} Org~A хэрэглэгчийн JWT-ээр Org~B-гийн өгөгдөлд хандах 6 хувилбар (cross-tenant SELECT/UPDATE, SQL injection, pending courier access, JWT forgery) бүгд амжилтгүй болсон (6/6). OTP brute-force туршилтад 5 дахь алдаатай оролдлогын дараа сесс блоклогдож, brute-force амжилтын онолын магадлал $5 \times 10^{-4}\%$ болж буурсан.

\textbf{Functional E2E.} Байгууллагын бүртгэл → бүтээгдэхүүн нэмэх → захиалга → даалгавар нийтлэх → хүргэгч claim → pickup → delivery → ePOD OTP → орлого бүртгэл гэсэн 8 алхамт урсгал зөв дарааллаар биелэв. Timestamp invariant (\texttt{assigned\_at} < \texttt{picked\_up\_at} < \texttt{delivered\_at} < \texttt{completed\_at}) зөрөлгүй, хүргэлтийн орлого автомат бүртгэгдсэн.

\textbf{Хязгаарлалт.} Туршилт нь Supabase Pro tier-ийн нөөцийн хязгаарт, 100 VU хүртэлх concurrent ачаалалд, synthetic өгөгдөл дээр, single-region тохиргоонд явагдсан. 500+ VU, multi-region, 3G/4G сүлжээний нөхцөл үнэлэгдээгүй. Бодит GPS tracking, QPay/SocialPay төлбөрийн integration, ML recommendation нь future work-т үлдсэн.

\textbf{Ерөнхий дүгнэлт.} Судалгааны гол дүгнэлт нь B2B хүргэлтийн платформын найдвартай байдлыг database түвшинд хэрэгжүүлсэн concurrency control (\texttt{FOR UPDATE SKIP LOCKED}) ба RLS гэсэн хоёр тулгуурт механизмд суурилан бүтээх боломжтойг туршилтын тоогоор нотолсон явдал юм. FCFS claim загвар, row-level isolation, bcrypt-д суурилсан ePOD OTP гэсэн гурван архитектурын шийдэл хамтдаа ажилласнаар multi-tenant орчинд race condition, cross-tenant access, OTP brute-force гэсэн гурван гол эрсдэлийг бодитоор шийдсэн.
